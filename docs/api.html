Stable touchpoints

These are the framework surfaces downstream projects are expected to depend on directly, and the ones maintainers should treat carefully when evolving the repository.

Public HTTP entrypoints

  • public/index.php
  • public/router.php
  • views/layouts/app.php as the shared HTML shell
  • public/vendor/fnlla-ui/ as the mandatory UI runtime bundle

Helper surface

  • asset() for public asset URLs
  • route() for named route generation
  • h() for escaped output
  • csrf_field() and csrf_token() for form protection
  • auth() for the auth manager access point

Middleware aliases

The router bootstrap registers a small and explicit middleware map, and route definitions build on top of those aliases.

Registered aliases

csrf
auth
authorize
cors
throttle

These are bound in bootstrap/router.php and are the supported names used in route definitions and groups.

Route-level helpers

$router->get("/dashboard", [AuthController::class, "dashboard"])
    ->middleware("auth")
    ->authorize("view-dashboard")
    ->name("dashboard");

authorize() and throttle() add metadata and the matching middleware behavior to the route definition.

CLI surface

The maintained console application covers project export, schema work, queueing, scheduling and repository maintenance routines.

Framework and runtime commands

  • php fnlla make:project
  • php fnlla fnlla-ui:sync
  • php fnlla fnlla-ui:validate
  • php fnlla route:list
  • php fnlla version:status and php fnlla version:sync

Data and background work

  • php fnlla migrate, migrate:rollback and migrate:status
  • php fnlla db:seed
  • php fnlla queue:work
  • php fnlla schedule:run
  • php fnlla cache:clear

Support files and machine-readable state

Versioning, ownership and runtime expectations are intentionally visible in repository metadata rather than hidden inside package registries.

State files

MANIFEST.json, README.md, VERSION and LICENSE.md should stay aligned whenever release metadata changes.

Validation scripts

scripts/validate-fnlla-ui.php, scripts/validate-version-manifest.php and scripts/build-docs.php --check are the core content-contract checks.

Source of truth

FNLLA PHP and FNLLA UI are maintained from GitHub repositories controlled by TechAyo LTD. Third-party package registries are outside the official maintainer workflow.
