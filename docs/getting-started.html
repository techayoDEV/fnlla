Quickstart

The fastest safe path is: keep this repo as the framework source, export a project for the real delivery, configure MySQL and validate the built-in runtime before building features.

Run the maintained framework locally

cd /workspace/fnlla
php -S 127.0.0.1:8080 -t public public/router.php

Open http://127.0.0.1:8080. For Apache-based hosting, set the document root to public/ and keep public/.htaccess in place.

Environment essentials

  • Copy .env.example to .env.
  • Set APP_URL.
  • Set DB_HOST, DB_PORT, DB_DATABASE, DB_USERNAME and DB_PASSWORD.
  • Set CONTACT_NOTIFICATION_EMAIL when using the contact flow.
  • The template starts with APP_ENV=development, APP_DEBUG=true and SESSION_SECURE=false so local HTTP on 127.0.0.1 works without session-cookie surprises.
  • Before production deployment, switch those values back to production-safe settings and serve the app over HTTPS.

Starter export workflow

php fnlla make:project ..\my-new-project "My New Project"

Use the exported directory as the real application repository. That keeps framework maintenance and project-specific delivery work cleanly separated.

Recommended first steps for a new downstream project

The exported project already runs, but the first pass should focus on wiring, runtime validation and replacing the demonstration surface with the actual product flow.

Initial commands

php fnlla fnlla-runtime:sync
php fnlla fnlla-runtime:validate
php fnlla framework:update --check --github
php scripts/test.php
php scripts/lint.php
php scripts/validate-version-manifest.php

First files to review

  • config/app.php
  • routes/web.php
  • src/Controllers/
  • views/pages/
  • public/assets/app.css
  • database/migrations/

When to stay in this repo

Direct work inside techayoDEV/fnlla is still the right move when the goal is framework maintenance rather than project delivery.

Good reasons

Hardening shared routing, auth, migrations, docs, scripts, source layout or the starter export itself.

Bad reasons

Building one client website directly in the framework repository and letting project-specific pages or schema pollute the shared base.
