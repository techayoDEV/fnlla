The runtime is built into FNLLA

FNLLA is not designed to be UI-agnostic in the official stack. The maintained application shell, shared layout and validation flow all assume that the built-in runtime is present as the only supported UI layer.

Where it lives

The built-in runtime lives under public/vendor/fnlla-runtime/. That directory is the local downstream copy used by the framework and by starter exports created from this repository.

Why it matters

Shared views, docs shells, theme switching, consent UI and application primitives all expect the shipped runtime classes, tokens and JavaScript behavior to exist locally.

How to use it in views

Downstream work should treat the runtime as part of FNLLA itself. When you build views, start from the shipped runtime primitives instead of introducing a second design system.

View-writing rules

  • Keep shared shell markup and asset loading in views/layouts/app.php.
  • Keep page body markup in views/pages/.
  • Build with the shipped section, container, stack, grid, card, btn and form primitives first.
  • Use public/assets/app.css only for project-specific refinements that the built-in runtime does not already cover.

Healthy starter shape

<section class="section">
  <div class="container">
    <div class="stack gap-lg">
      <header class="stack gap-sm">...</header>
      <div class="grid grid-2 gap-md">...</div>
    </div>
  </div>
</section>

That is the default mental model: page content in views/pages/, global shell in the layout, runtime classes for structure, then narrow project styling where needed.

Supported runtime boundary

The runtime is part of FNLLA, not a replaceable add-on.

Supported

  • Keep the built-in runtime under public/vendor/fnlla-runtime/.
  • Sync updates through the official TechAyo-maintained runtime workflow.
  • Build pages with the shipped runtime layout, card, button, alert and form primitives.

Unsupported

  • Replacing the built-in runtime with Tailwind, Bootstrap or another framework in the official stack.
  • Loading third-party UI assets from random CDNs instead of the built-in runtime.
  • Changing shared shell markup so the framework can no longer validate the runtime contract.

Practical consequence

FNLLA docs, starter pages and application shell all assume the same runtime classes and behavior, so view work stays aligned when you build with those primitives from the start.

Full starter shell syntax

The built-in runtime is meant to be used through the full FNLLA starter shell, not only through isolated page fragments. The real baseline is the shared layout in views/layouts/app.php plus page content rendered inside its main area.

Shared starter layout

<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <meta name="theme-color" content="#1A4137">
  <title><?= h($pageMeta["title"]) ?></title>
  <link rel="stylesheet" href="<?= h(asset("vendor/fnlla-runtime/assets/css/fnlla-runtime.css")) ?>">
  <link rel="stylesheet" href="<?= h(asset("assets/app.css")) ?>">
</head>
<body data-fnlla-theme="default">
  <header>
    <section class="section">
      <div class="container">
        Header / Navigation
      </div>
    </section>
  </header>

  <main>
    <?= $content ?>
  </main>

  <footer>
    <section class="section">
      <div class="container">
        Footer
      </div>
    </section>
  </footer>

  <script src="<?= h(asset("vendor/fnlla-runtime/assets/js/fnlla-runtime.js")) ?>"></script>
</body>
</html>

This is the real starter baseline: runtime assets in head, shared navigation in header, page injection through <?= $content ?> inside main, then a simple footer and runtime JavaScript at the end of body.

Sync and validation workflow

When the maintained runtime source changes, update the built-in runtime deliberately and validate it before treating the repository as release-ready.

Recommended commands

php fnlla fnlla-runtime:sync
php fnlla fnlla-runtime:validate
php scripts/build-docs.php --check

Use the explicit sync command when you want to pull in maintained runtime changes. The timed development guard keeps local state fresh and can auto-repair a missing built-in runtime, while these commands remain the deliberate maintainer workflow.

Operational rule

If the maintained runtime changes upstream, sync the vendored copy before shipping FNLLA changes that rely on the new runtime behavior or styles.

Starter-export consequence

A new project created from FNLLA inherits the built-in runtime immediately, so downstream teams can build pages without adding a separate UI package step.
