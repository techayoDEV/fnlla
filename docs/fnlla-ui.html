FNLLA UI is a required runtime dependency

FNLLA PHP is not designed to be UI-agnostic in the official stack. The maintained application shell, shared layout and validation flow all assume that FNLLA UI is present as the only supported UI runtime.

Where it lives

The vendored runtime lives under public/vendor/fnlla-ui/. That directory is the local downstream copy used by the framework and by starter exports created from this repository.

Why it matters

Shared views, docs shells, theme switching, consent UI and application primitives all expect FNLLA UI classes, tokens and JavaScript behavior to exist locally.

Supported dependency contract

Downstream work should treat FNLLA UI as a first-class product dependency, not as a replaceable design preference.

Supported

  • Keep the vendored runtime under public/vendor/fnlla-ui/.
  • Sync updates from the official FNLLA UI repository workflow.
  • Build pages with FNLLA UI layout, card, button, alert and form primitives.

Unsupported

  • Replacing FNLLA UI with Tailwind, Bootstrap or another framework in the official stack.
  • Loading third-party UI assets from random CDNs instead of the vendored runtime.
  • Changing shared shell markup so the framework can no longer validate the UI contract.

Sync and validation workflow

When FNLLA UI changes upstream, update the vendored runtime deliberately and validate it before treating the repository as release-ready.

Recommended commands

php fnlla fnlla-ui:sync
php fnlla fnlla-ui:validate
php scripts/build-docs.php --check

These commands keep the local downstream copy aligned and confirm that docs and application shell expectations still match the vendored runtime.

Operational rule

If you change FNLLA UI upstream, sync the vendored copy before shipping FNLLA PHP changes that rely on the new runtime behavior or styles.

Starter-export consequence

A new project created from FNLLA PHP inherits the vendored FNLLA UI runtime immediately, so downstream teams can build pages without adding a separate UI package step.
