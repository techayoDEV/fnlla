Repository and runtime boundary

FNLLA PHP is maintained as a full repository, but downstream consumers should only treat a smaller subset as the supported application runtime surface.

Supported downstream runtime surface

  • public/index.php and public/router.php as HTTP entrypoints.
  • public/assets/ for project-owned public assets.
  • public/vendor/fnlla-web/ for the vendored authoritative UI runtime.
  • views/layouts/ and views/pages/ as the server-rendered delivery layer.

Maintainer-owned internals

  • bootstrap/, config/, routes/ and src/ define framework behavior.
  • database/ and tests/ keep schema, seed and repository-level verification logic.
  • scripts/ owns sync, validation, lint and docs-generation routines.
  • MANIFEST.json, VERSION, README.md and LICENSE.md are release state files, not app content.

What ships together

A downstream project exported from this repository receives a working application base rather than a bare framework package.

FNLLA Web dependency boundary

FNLLA PHP is not UI-agnostic in the official stack. The UI runtime is a first-class dependency, not an optional styling preference.

Supported rule

Keep the vendored FNLLA Web runtime under public/vendor/fnlla-web/ and sync it only from the official GitHub source of truth.

Unsupported rule

Do not swap in Tailwind, Bootstrap, Bulma, Foundation, UIkit, Materialize or Semantic UI in the official framework stack.

Operational consequence

Bootstrap, validation and the shared application layout all assume the FNLLA Web shell structure and class families stay intact.
