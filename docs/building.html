Build with the grain of the framework

FNLLA stays fast and legible when delivery work follows the small request lifecycle that already exists in the repository instead of layering new abstractions too early.

1. Request

Entrypoints

public/index.php and public/router.php are the maintained HTTP entrypoints.

2. Bootstrap

Bootstrap and router

bootstrap/app.php, bootstrap/common.php and bootstrap/router.php wire environment, services, middleware aliases and route resolution.

3. Delivery

Controller or closure

A controller action or a small closure builds the response payload and decides whether the output is HTML, JSON or a redirect.

4. Response

View or API output

views/layouts/app.php owns the HTML shell, while Response::json() and JsonResource cover structured API output.

Page-building pattern

New product surfaces usually land in four moves: route, controller, view and optional persistence or auth rules.

Route and controller

$router->get("/services", [PageController::class, "services"])->name("services");

public function services(Request $request): Response
{
    return $this->view("pages/services", [
        "pageTitle" => "Services",
    ]);
}

View structure

<section class="section">
  <div class="container">
    <div class="grid grid-2 gap-md">...</div>
  </div>
</section>

Keep page templates under views/pages/, let the shared layout own the document shell and shared runtime assets, and compose the page with built-in runtime classes first.

Built-in runtime first

When you write a new view, think in terms of the runtime FNLLA already ships rather than a second front-end layer you add later.

Reach for these first

  • section and container for page framing
  • stack and grid for spacing and layout
  • card, badge and btn for common UI blocks
  • input, select and textarea for forms

Avoid early drift

  • Do not load a second CSS framework.
  • Do not rebuild the global shell inside a page template.
  • Do not bypass runtime tokens when the existing spacing, color or typography scale already fits the need.

Forms, validation and redirects

The framework is happiest when forms follow a predictable GET page + POST submit + validation + redirect pattern.

Form rules

  • Submit to a named POST route.
  • Use csrf_field() in every state-changing form.
  • Validate before side effects.
  • Flash errors or success state back into the next request.
  • Redirect after success instead of rendering directly from the POST.

Validation example

$payload = [
    "email" => trim((string) $request->input("email", "")),
];

$this->validate($payload, [
    "email" => ["required", "email", "max:160"],
]);

Persistence and protected areas

Use MySQL for official projects, keep query logic out of views and rely on middleware plus authorization gates for protected sections.

Database workflow

Create migrations under database/migrations/, run them with php fnlla migrate and keep repeated data access in controllers or dedicated services.

Protected routes

Use auth middleware for identity checks and authorize() or route-level authorization for capabilities such as admin access.
